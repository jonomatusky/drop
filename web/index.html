A static-artifact publishing pipeline

How drop.jonomatus.ky works

One command turns a local HTML file, report, or screenshot bundle into a clean public URL on the open internet — with a password gate, expiry, revocation, and an audit trail behind it. Here is the whole machine.

drop.jonomatus.ky/<slug>/

What this is: a personal, self-hosted tool built by jonomatusky and run on his own Cloudflare account. It is not a product you can sign up for — this page just explains the design, because the design has a few details worth showing. (This very page was published with the tool it describes.) The code is open source: github.com/jonomatusky/drop.

Architecture

A thin CLI and a small Worker

The system is two halves with a clean seam between them. A local command-line tool does all the work that needs your machine — scanning, hashing, uploading. A Cloudflare Worker at the edge does all the work that needs to be live 24/7 — serving pages, checking passwords, honoring expiry. Neither half holds more power than its job requires.

Local · the CLI

drop path/to/artifact

Runs a no-leak preflight, resolves a readable slug, uploads each file as an object to R2 storage, then calls the Worker's admin API to write the canonical manifest for the share.

↓
Object storage · R2

Files land in drop-artifacts

Cloudflare R2 holds the raw bytes of every published file. The CLI writes them; the Worker reads them. Nothing is served from your machine.

↓
Edge · the Worker

Serves /<slug>/… from R2

On every request it looks up the manifest in a D1 database, enforces the auth mode, checks expiry and revocation, sets no-index headers, and streams the object back — logging the access.

The split is also a security boundary. The deployed Worker carries no API token at all — it reaches R2 and the database through Cloudflare bindings, so no broad credential is ever left sitting on the edge.

State

R2 for the bytes, D1 for the record

Storage and bookkeeping are kept separate. R2 is the object store — it holds the actual file contents. D1 (Cloudflare's edge SQLite) holds the metadata: one row per share and an append-only audit log.

The share manifest

Each share is a row: its slug, auth mode, content hash, file list, byte count, the preflight scan result, plus created / expiry / revoked timestamps and who uploaded it.

The audit trail

A separate audit_events table records what happened to each share over time — creation, access, revocation — as an immutable, append-only history keyed by slug.

Because the manifest carries an expiry and a revocation timestamp, a share can be switched off (drop revoke <slug>) or made to lapse without touching the stored files — the Worker simply refuses to serve a slug the database says is dead.

Naming

Readable slugs, no random IDs

Shares get short, human-readable URLs — drop.jonomatus.ky/ted-page/, not a wall of hex. The slug comes from a --client or --label you pass, or, if you pass neither, the artifact's own filename (report.html → report). It is lowercased, hyphen-joined, and capped at 40 characters.

The interesting part is how name collisions are handled: at publish time, not with a random suffix. The CLI asks the Worker for the list of existing shares and takes the readable name if it's free. If it's taken, it appends -2, -3, and so on — ted-page-2.

  • Every existing share counts as taken — including expired and revoked ones — so a new artifact never silently overwrites a name that was used before.
  • The Worker stores by slug (an upsert), which is exactly why the CLI owns clash-avoidance: the edge would happily overwrite, so the client makes sure it never has to.
  • The result is URLs you can read aloud, with no UUID noise — and no accidental clobbering.

Access

Three visibility tiers

Borrowing YouTube's model, a share is one of three things. To the Worker there are really only two auth modes — password and public — and the third tier is purely a difference in how the slug is shaped.

Private · default (no flag)

Password-protected. The slug is just a label; the password is the lock, so a guessable name leaks nothing. The password is verified server-side and sent out-of-band. This is what you get if you pass no visibility flag at all.

drop.jonomatus.ky/acme-proposal/ + password
Unlisted --unlisted

No password, but the slug gets an unguessable ~82-bit random tail. The URL is the capability — anyone with the link gets in, nobody can enumerate their way to it. Strong entropy matters here because the Worker only rate-limits failed password attempts, so a no-password share isn't throttled.

drop.jonomatus.ky/acme-proposal-k7f3m9q2x8p4n6tz/
Public --public

No password, clean guessable slug — for content meant to be found and shared. To the Worker it's identical to unlisted (auth_mode="public"); the only difference is the short, readable slug. This page was published this way.

The load-bearing detail

A bucket-scoped R2 token, over S3

The most deliberate design choice is about blast radius. The Cloudflare account behind this also runs other things — another app's storage, databases, and Workers. So the credential that drop uses for its constant job — uploading files on every run — is scoped down to exactly one bucket and nothing else.

That scoping forces an unusual technical path, and it's worth understanding why: 

# The obvious command won't do:
wrangler r2 object put …
  ↳ hits Cloudflare's REST API
  ↳ which only accepts an R2 *Admin* token
  ↳ and Admin tokens are account-wide — they CANNOT be
    scoped to a single bucket.

# So the CLI uploads a different way:
S3 PUT (SigV4) with an Object Read & Write token
  ↳ Object R/W tokens ARE bucket-scopeable
  ↳ their credential is an S3 Access Key ID + Secret
  ↳ scoped to drop-artifacts only

In short: the only Cloudflare token type that can be locked to a single bucket is an Object Read & Write token, and that token speaks S3, not Cloudflare's REST API. So the CLI uploads with an S3 SigV4 PUT — trading the convenient wrangler command for a credential that simply cannot reach anything it shouldn't.

✓ Can

Write objects into drop-artifacts — the one bucket this tool owns.

✗ Cannot

Touch any other bucket, database, or Worker on the account. A leak of this key exposes only published artifacts.

There's even a self-test for it. drop scope-test uses the upload key to prove it can write the artifacts bucket and is denied (401/403) on a known forbidden bucket. If the forbidden bucket is ever readable, the token is mis-scoped and the check fails loudly.

Before anything leaves the machine

A no-leak preflight

Publishing is one-way — public, forwardable, cacheable, and able to outlive a delete. So every run is gated by a local scan before a single byte is uploaded:

  • Secret scan. A gitleaks content pass blocks anything that looks like a key or token.
  • Dangerous filenames. .env, private keys, and credential paths are refused by name.
  • PII. High-confidence personal data blocks unless explicitly overridden with a logged reason.
  • Images. OCR-scanned when tooling is available; otherwise you must accept the limitation after a manual look.
  • Trackers & archives. External scripts and tracking pixels warn; archive files are blocked outright.
  • Final confirmation. A last prompt reminds you the artifact is public and may survive deletion.

Passwords, when used, are hashed with PBKDF2 at 100,000 iterations — the ceiling Cloudflare's WebCrypto allows in a Worker — and the CLI and Worker are kept in lockstep on that count so a verifier minted on one side always checks out on the other.

In one breath

The whole pipeline

You run drop on a file. It scans for leaks, picks a clean unique slug, uploads the bytes to a single bucket using a key that can't reach anything else, and registers the share with an edge Worker. From then on, that Worker serves your artifact from a short public URL — enforcing the password, the expiry, and the revocation switch on every request, and writing every access to an audit log. Two small pieces, one clean seam, and a credential scoped so tightly that the worst-case leak exposes only the things you already chose to publish.
